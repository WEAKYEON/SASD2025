## a) ความหมายของ Comments (Code Smell)
การมีคอมเมนต์ในโค้ด (Comments) ฟังดูเหมือนเป็นเรื่องที่ดี แต่ในมุมมองของ Code Smell มักเป็นสัญญาณเตือนว่า **"โค้ดชุดนั้นเขียนไว้ไม่ดี หรือซับซ้อนเกินไปจนต้องเขียนคำอธิบายกำกับ"**

หลักการสำคัญคือ *Code should explain itself* (โค้ดควรอธิบายตัวมันเองได้) หากเราต้องเขียนคอมเมนต์เพื่ออธิบายว่าโค้ดบรรทัดนี้ทำงานอย่างไร แสดงว่าเราควรปรับปรุงการตั้งชื่อตัวแปรหรือฟังก์ชันให้สื่อความหมายชัดเจนขึ้น แทนที่จะใช้คอมเมนต์มาช่วยอธิบาย (Deodorant)

## b) ตัวอย่างโค้ดที่เกิด Code Smell
ปัญหามักเกิดจากการตั้งชื่อตัวแปรที่ไม่สื่อความหมาย หรือ Logic ที่ซับซ้อนจนต้องเขียนคอมเมนต์กำกับไว้

```java
public void check(int s) {
    // ตรวจสอบว่านักเรียนมีสิทธิ์สอบหรือไม่ (คะแนนต้องมากกว่า 50)
    if (s > 50) {
        System.out.println("Pass");
    }
}
```
ปัญหา: ใช้ชื่อตัวแปร s ซึ่งไม่สื่อความหมาย ทำให้ต้องเสียเวลาเขียนคอมเมนต์อธิบายว่า s คือคะแนน และเงื่อนไข > 50 คือเกณฑ์ผ่าน

## c) ตัวอย่างโค้ดที่ Refactoring แล้ว
ใช้วิธี Rename Variable และ Extract Method เพื่อให้ชื่อของตัวแปรและฟังก์ชันทำหน้าที่อธิบายโค้ดแทนคอมเมนต์

```Java
public void checkStudentEligibility(int studentScore) {
    if (isEligibleForExam(studentScore)) {
        System.out.println("Pass");
    }
}

private boolean isEligibleForExam(int score) {
    return score > 50;
}
```
สิ่งที่แก้ไข: เปลี่ยน s เป็น studentScore และแยกเงื่อนไขออกมาเป็นฟังก์ชัน isEligibleForExam ทำให้ผู้อ่านเข้าใจได้ทันทีว่าโค้ดนี้ทำอะไรโดยไม่ต้องอ่านคอมเมนต์

## d) กรณีที่ Code Smell นี้ไม่ได้บ่งบอกถึงโค้ดที่มีปัญหา
ไม่ใช่ทุกคอมเมนต์ที่เป็นสิ่งไม่ดี บางกรณีเราจำเป็นต้องคงคอมเมนต์ไว้ เช่น:
- อธิบายเหตุผล: เมื่อต้องการอธิบายว่า "ทำไม" ถึงเขียนโค้ดแบบนี้ (เช่น เป็นข้อจำกัดทางธุรกิจ หรือต้องเขียนอ้อมๆ เพื่อแก้บั๊กของ Library)
- API Documentation: การเขียน JavaDoc หรือคำอธิบายฟังก์ชันเพื่อให้ Developer คนอื่นรู้วิธีเรียกใช้งาน
